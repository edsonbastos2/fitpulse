# Tarefa 2.0: Migration do Schema de Roles

<critical>Ler os arquivos de prd.md e techspec.md nesta pasta. Se você não ler esses arquivos, sua tarefa será invalidada.</critical>

## Visão Geral

Criar a migração SQL que estabelece toda a estrutura de banco de dados para o sistema de roles: tabelas `roles`, `user_roles`, `role_permissions`, `trainer_students`, dados iniciais (seed), RLS policies para isolamento de dados, e migração automática de usuários existentes para o role `user`.

### Conformidade com Skills Padrões

- **`supabase-postgres-best-practices`** — RLS policies, indexação, design de schema, performance de queries
</skills>

<requirements>

### RF-01
O sistema deve armazen roles de usuário em tabela associativa (`user_roles`) vinculada ao perfil do usuário

### RF-02
Cada usuário pode possuir múltiplos roles simultaneamente

### RF-17
Um Personal Trainer deve visualizar apenas dados dos alunos vinculados a ele

### RF-19
O Superadmin deve ter acesso de leitura a todos os dados da plataforma para fins de auditoria

### RF-20
O sistema deve impedir que um usuário acesse dados de outro usuário sem vínculo explícito

</requirements>

## Subtarefas

- [ ] 2.1 Criar arquivo `supabase/migrations/001_roles_and_permissions.sql`
- [ ] 2.2 Criar tabela `roles` (catálogo) com colunas: id, slug, name, description, created_at
- [ ] 2.3 Criar tabela `user_roles` (associativa M:N) com unique(user_id, role_id) e FKs com ON DELETE CASCADE
- [ ] 2.4 Criar tabela `role_permissions` com unique(role_id, permission)
- [ ] 2.5 Criar tabela `trainer_students` com unique(trainer_id, student_id) e CHECK(trainer_id != student_id)
- [ ] 2.6 Inserir dados seed: 3 roles (user, personal_trainer, superadmin)
- [ ] 2.7 Inserir permissões para cada role conforme `ROLE_PERMISSIONS` na techspec
- [ ] 2.8 Inserir role `user` para todos os usuários existentes (idempotente)
- [ ] 2.9 Criar RLS policies para `user_roles` (select own, manage by self/superadmin)
- [ ] 2.10 Criar RLS policies para `roles` e `role_permissions` (select all)
- [ ] 2.11 Criar RLS policies para `trainer_students` (select by trainer/student/superadmin, manage by trainer/superadmin)
- [ ] 2.12 Criar RLS policies estendidas para `user_profiles` (select own/by-PT/by-superadmin, update/insert own)
- [ ] 2.13 Aplicar migration localmente com `npx supabase db push` e verificar sem erros
- [ ] 2.14 Verificar que todos os usuários existentes receberam role `user`
