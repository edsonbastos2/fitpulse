# Tarefa 9.0: Server Middleware de Validação de Roles

<critical>Ler os arquivos de prd.md e techspec.md nesta pasta. Se você não ler esses arquivos, sua tarefa será invalidada.</critical>

## Visão Geral

Criar o server middleware `server/middleware/role-validation.ts` que valida roles em endpoints de API (server routes Nuxt). Esta é a camada de proteção backend do sistema defense-in-depth: mesmo que o frontend middleware seja bypassado, o servidor rejeitará requisições de usuários sem os roles adequados.

## Detalhes de Implementação

Consultar a **Tech Spec** nas seções:
- **Endpoints de API (Server Routes)** → código de exemplo de `server/middleware/role-validation.ts`

O middleware roda em **toda** request do server. Deve ser eficiente:
1. Check rápido se URL começa com prefixo protegido
2. Se não, retornar imediatamente (sem query ao banco)
3. Se sim, extrair user e verificar roles

Para extrair o user_id do contexto Supabase no server, usar `getCookie(event, 'sb-...-auth-token')` e decodificar o JWT, ou usar `useSupabaseClient()` server-side.

## Critérios de Sucesso

- [ ] `server/middleware/role-validation.ts` existe e é registrado automaticamente pelo Nuxt
- [ ] GET `/api/students` com role `user` → retorna 403
- [ ] GET `/api/students` com role `personal_trainer` → permite passar (200 ou próximo handler)
- [ ] GET `/api/admin/anything` com role `personal_trainer` → retorna 403
- [ ] GET `/api/admin/anything` com role `superadmin` → permite passar
- [ ] GET `/api/workouts` (rota não protegida) → permite passar sem verificação de role
- [ ] Requisição sem user_id (não autenticado) → retorna 401
- [ ] Log `console.warn` é emitido para cada 403
