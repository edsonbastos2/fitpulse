# ADR-004: Feature-Based Directory Structure for Workout Module

## Status

Accepted

## Date

2026-04-13

## Context

The workout module introduces 6 new pages (list, create, detail, start, history, session detail) and numerous supporting components. The existing FitPulse codebase uses a flat component structure (`components/ui/`, `components/catalog/`) which works for small features but becomes unwieldy for a module of this size. Clear organization boundaries are needed to maintain code navigability.

## Decision

Organize the workout module using **feature-based subdirectories** with separate folders for pages, components, and composables:

```
pages/workouts/          # Route pages (Nuxt file-based routing)
components/workouts/     # Module-specific Vue components
composables/             # Module composables at root level (project convention)
stores/                  # Pinia stores (new directory)
```

Within `components/workouts/`, group by sub-feature:
```
components/workouts/
  create/                # Workout creation components
    TemplateSelector.vue
    WorkoutForm.vue
    ExercisePicker.vue
  list/                  # Workout list components
    WorkoutList.vue
    WorkoutListItem.vue
    WorkoutEmptyState.vue
  session/               # Session execution components
    SessionRunner.vue
    SetLogger.vue
    RestTimer.vue
    SessionComplete.vue
  history/               # Session history components
    SessionList.vue
    SessionListItem.vue
    SessionDetail.vue
    InsightCard.vue
```

## Alternatives Considered

### Alternative 1: Flat pages only

- **Description**: Each route is a single `.vue` file in `pages/workouts/` with all logic inline. No supporting component files.
- **Pros**: Simple; no new directories; Nuxt auto-routing works out of the box
- **Cons**: Pages become 400+ lines; components untestable in isolation; duplicated UI patterns across pages
- **Why rejected**: Module is too large for flat pages; violates single responsibility

### Alternative 2: Feature directory grouping all artifacts together

- **Description**: Each feature has its own directory: `workouts/create/` contains page, components, composables, and tests together.
- **Pros**: Maximum cohesion; easy to find all files for a feature
- **Cons**: Breaks Nuxt file-based routing conventions; non-standard for Vue/Nuxt projects; conflicts with existing project structure
- **Why rejected**: Conflicts with established Nuxt conventions; would require custom routing configuration

## Consequences

### Positive

- Clear separation between pages (routing), components (UI), and composables/stores (logic)
- Consistent with existing FitPulse patterns (`components/ui/`, `components/catalog/`)
- Each component is testable in isolation with Vue Test Utils
- Easy to locate files: UI → `components/workouts/`, logic → `composables/` or `stores/`, routes → `pages/workouts/`
- Scales well — additional features can add subdirectories without restructuring

### Negative

- More files and directories to navigate (15+ new files vs 5 flat pages)
- Requires discipline to maintain boundaries (e.g., not putting business logic in page files)
- `components/workouts/` adds ~4KB to auto-import overhead (negligible)

### Risks

- **Nuxt auto-import**: Components in `components/workouts/` are auto-imported by Nuxt's component scanner (configured with `pathPrefix: false`). No risk — this is the existing pattern.
- **Naming collisions**: Components like `WorkoutForm` could conflict with future modules. Mitigation: prefix with feature scope if needed (already done: `TemplateSelector` is unique enough).

## Implementation Notes

- The `components` config in `nuxt.config.ts` already includes `{ path: '~/components', pathPrefix: false }`, so `components/workouts/` will be auto-discovered without additional configuration.
- Page files in `pages/workouts/` follow Nuxt file-based routing: `index.vue` → `/workouts`, `create.vue` → `/workouts/create`, `[id].vue` → `/workouts/:id`, `[id]/start.vue` → `/workouts/:id/start`.
- Composables (`useSessionRunner`, `useInsights`) live at `composables/` root level following existing project convention.
- Pinia stores live in new `stores/` directory (not yet present in project).
