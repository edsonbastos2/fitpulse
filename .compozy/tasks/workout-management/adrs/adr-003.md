# ADR-003: Pinia Store for Session State Management

## Status

Accepted

## Date

2026-04-13

## Context

The workout session execution page (`/workouts/[id]/start`) requires managing complex state: active exercise index, individual set logs, rest timer countdown, auto-fill data from previous sessions, localStorage checkpoint for crash recovery, and session summary computation. The composable-only approach used elsewhere in FitPulse (via `useState`) would result in scattered state variables with unclear boundaries.

## Decision

Use **Pinia** as the state management solution for session-related state. Create a dedicated store (`stores/session.ts`) with explicit state properties, actions, and getters. Install `pinia` and `@pinia/nuxt` as project dependencies. The store manages: active workout session ID, exercise list with logs, current exercise/set index, timer state, checkpoint data, and session summary.

## Alternatives Considered

### Alternative 1: Composable `useSessionRunner` only

- **Description**: Dedicated composable encapsulating all session state with `useState()` and `ref()` internally.
- **Pros**: No new dependency; consistent with existing FitPulse patterns (all composables use `useState`)
- **Cons**: State spread across multiple `useState` calls; harder to persist/restore checkpoint; less explicit boundary between session state and component-local state
- **Why rejected**: Session state is complex enough to warrant structured store with actions, getters, and explicit state shape

### Alternative 2: `useState()` directly in page component

- **Description**: Multiple `useState()` calls directly in the page component without abstraction layer.
- **Pros**: Simplest approach; zero new files or dependencies
- **Cons**: Page component becomes monolithic; state logic untestable in isolation; no reusability across components
- **Why rejected**: Violates separation of concerns; makes the session runner page 400+ lines of mixed UI and state logic

## Consequences

### Positive

- Explicit, testable state management with clear actions and getters
- Pinia DevTools available for debugging during development
- Scalable pattern — additional stores can be added for workouts, insights, etc.
- Session state is serializable for localStorage checkpoint
- Actions encapsulate complex logic (auto-fill, checkpoint save/restore, session summary)

### Negative

- New dependency (pinia, @pinia/nuxt) to install and configure
- Slight learning curve if team is only familiar with Vue composables
- Adds ~5KB to bundle size (minified)

### Risks

- **Integration with `@nuxtjs/supabase`**: Pinia stores need access to Supabase client. Mitigation: use `useSupabaseClient()` inside store actions (Pinia supports Nuxt composables via `@pinia/nuxt` module).
- **SSR concerns**: Pinia defaults to SSR-compatible state. Mitigation: session runner page uses `ssr: false` or guards to prevent server-side execution.

## Implementation Notes

1. Run `pnpm add pinia @pinia/nuxt`
2. Add `@pinia/nuxt` to `modules` array in `nuxt.config.ts`
3. Create `stores/session.ts` with `defineStore('session', { state, actions, getters })`
4. Store state is session-scoped — do not persist across route navigations unless explicitly checkpointed to localStorage
5. The store should NOT replace `useWorkouts` composable — they serve different purposes (useWorkouts = CRUD + API, Pinia session store = real-time execution state)
